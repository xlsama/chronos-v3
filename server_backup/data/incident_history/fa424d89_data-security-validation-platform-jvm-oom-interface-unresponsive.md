# 事件排查报告

## 事件概要
- **标题**: 数据安全校验平台接口无响应（JVM OOM）
- **严重程度**: P3
- **处理状态**: 已解决

## 问题描述
**受影响服务**: 数据安全校验平台 (ID: `1054f9cf-407c-4aba-b02a-3a8a95bc5a29`)
**症状**: 外部业务应用无法通过接口提交数据或查询校验结果，所有 HTTP 请求超时，表现为接口无响应。
**影响范围**: 依赖该平台的内部多应用业务系统无法进行数据合规校验，可能导致业务数据流转阻塞。

## 排查过程
1. **上下文收集与假设构建**:
   - 检索知识库确认项目为“数据安全校验平台”，涉及接口包括 `/` (POST) 和 `/api/check/data` (POST)。
   - 由于缺乏详细服务拓扑信息，构建三个排查假设：H1 服务进程崩溃/假死、H2 基础资源耗尽、H3 网络连通性问题。

2. **服务进程与资源检查 (验证 H1)**:
   - 登录服务器 `10.200.100.85` (容器宿主机)，执行 `ps aux` 发现 Java 进程 (`java -jar app.jar`, PID 18718) 仍在运行，但 CPU 占用率较高。
   - 检查端口监听状态 (`netstat`)，确认 8082 端口处于 `LISTEN` 状态。
   - 执行本地 `curl` 测试接口，发现连接超时 (10秒)，确认服务虽存活但无法处理请求。

3. **容器与日志分析**:
   - 确认应用运行在 Docker 容器 `yum-data-security` 中 (镜像 `yum-data-security:dev`)。
   - 查看容器最近日志 (`docker logs --tail 200`)，发现关键报错：`java.lang.OutOfMemoryError: Java heap space`。
   - 日志显示异常发生在 Spring MVC 请求处理链中，伴随 `NestedServletException`，证实因堆内存不足导致服务假死。

4. **修复操作**:
   - 执行容器重启命令：`sudo docker restart yum-data-security`。

5. **验证恢复**:
   - 等待 5 秒后再次测试接口。
   - `curl http://localhost:8082/health` 返回 `404` (路径不存在但连接正常)。
   - `curl http://localhost:8082/` 返回 `HTTP 200`，响应时间 `0.020s`。
   - 确认服务恢复正常。

## 根因分析
根本原因为 **JVM 堆内存溢出 (Java Heap Space Out of Memory)**。
- 容器 `yum-data-security` 内的 Java 应用在运行过程中，堆内存使用达到上限，触发 `OutOfMemoryError`。
- 此时 JVM 线程池可能已满或无法分配新线程处理请求，导致虽然进程未退出且端口保持监听，但所有 incoming HTTP 请求均无法得到响应，表现为“接口无响应”。
- 故障持续时间约为 23 小时（自上次启动以来），表明存在潜在的内存泄漏或长期高负载下的内存配置不足问题。

## 修复措施
1. **紧急恢复**:
   - 执行 `sudo docker restart yum-data-security` 重启容器，释放内存并重新加载应用，服务即刻恢复可用。

2. **后续优化建议 (非本次执行，需跟进)**:
   - **调整 JVM 参数**: 检查并增加容器的 `-Xmx` (最大堆内存) 配置，建议设置为容器内存限制的 70%-80%。
   - **监控告警**: 部署 JVM 内存监控及 GC 频率告警，设置阈值预警。
   - **内存分析**: 导出 OOM 时的 Heap Dump 文件，使用 MAT 等工具分析是否存在内存泄漏点（如大对象未释放、缓存无限增长）。
   - **健康检查**: 完善 Docker/K8s 的健康检查机制，实现故障自动检测与自愈。
