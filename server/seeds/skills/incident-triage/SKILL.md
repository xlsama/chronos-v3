## 适用场景

当你刚接到一个事件，还不能确定根因时，优先使用本技能。它不是某类故障的专项剧本，而是所有故障都通用的分诊框架。

常见触发场景：

- 页面报错、接口无响应、超时、pending
- 业务慢、CPU 高、内存高、磁盘满
- 数据不一致、查询失败、写入失败
- 服务发布后异常、配置修改后异常
- 用户只给了截图、日志片段或一句现象描述

## 执行环境说明

你是一个远程运维 Agent，不在目标服务器上运行。所有操作通过以下工具完成：

- **`ssh_bash(server_id, command)`** — 在远程目标服务器执行命令
- **`service_exec(service_id, command)`** — 直连已注册服务执行查询
- **`bash(command)`** — 本地辅助命令，例如 curl、文本处理
- **`list_servers()`** — 获取可 SSH 的服务器
- **`list_services()`** — 获取已注册服务

## 第一步：症状归类

先回答“用户现在看到什么”，不要先回答“为什么会这样”。

把问题先归到一个或多个症状桶里：

- 可用性：无响应、超时、5xx、健康检查失败
- 性能：慢、延迟高、QPS 掉、CPU 高、内存高
- 数据正确性：查不到、重复、错乱、延迟
- 依赖异常：数据库、缓存、MQ、第三方不可用
- 变更异常：发布、配置、镜像、证书、权限变更后异常
- 安全/网络：DNS、证书、防火墙、ACL、鉴权失败
- 资源异常：磁盘满、FD 耗尽、线程爆炸、OOM

如果用户只给了截图或一句描述，先把截图/附件里的文本线索整理成：

- 受影响对象
- 错误现象
- 关键时间
- 关键标识（域名、端口、服务名、报错码）

## 第二步：目标锁定

先缩小到候选项目、服务、服务器和依赖链，避免盲查。

推荐顺序：

1. 看知识库上下文里的项目候选、AGENTS.md、业务背景、入口线索
2. 如果不够，再调用 `list_servers()` 和 `list_services()`
3. 结合事件描述，整理出：
   - 候选项目
   - 候选服务
   - 候选服务器
   - 关键依赖

如果当前命中了多个项目，不要立刻选一个；先找更能区分目标的线索，例如：

- 域名 / URL / 端口
- 服务名 / 容器名 / 进程名
- 服务器 IP / 环境名 / 机房标识
- 数据库名 / Redis key 前缀 / Topic 名称

## 第三步：选择首个观测面

不要固定套某一条命令链。根据症状，选择最贴近现象的第一层验证。

首轮探测的写法规则：

- 不要在第一次失败验证时使用 `2>/dev/null` 或 `>/dev/null` 吞掉原始错误
- 不要用会掩盖左侧失败的管道写法去判断“命令不存在”或“服务没起”
- HTTP 探测优先使用 `curl -sS`，保留错误信息；不要只拿 `000` 就下结论
- 只有在确认命令可用、只是为了降噪时，第二轮才允许按需压 stderr

### 外部探测面

适用：接口无响应、超时、健康检查失败、页面报错

- `curl`
- TCP 端口探测
- HTTP 状态码 / 耗时验证

### 运行面

适用：怀疑服务挂掉、假活、阻塞、资源耗尽

- 进程 / systemd / supervisor / pm2
- Docker / containerd / 容器状态
- CPU / 内存 / 磁盘 / 监听端口

### 依赖面

适用：报错像数据库、缓存、MQ、第三方问题

- `service_exec` 验证依赖是否可连、是否报错、是否超时

### 日志面

适用：需要确认具体错误栈、OOM、CrashLoop、权限异常

- 应用日志
- 容器日志
- 系统日志

### 变更面

适用：故障与发布、配置、镜像、证书变更时间强相关

- 最近发布
- 配置变更
- 镜像版本 / rollout 历史

## 第四步：收敛故障域

用证据把问题收敛到一个较小的故障域，再决定是否切专项技能。

几个通用判断原则：

- 依赖探测成功，只能说明依赖可连，不等于业务应用健康
- 未锁定项目、服务、服务器前，不要直接建议重启或回滚
- 没有容器证据时，不要机械进入 Docker 诊断
- 有容器证据时，优先切到 `docker-container`，不要先把 JVM / Node 裸进程当默认根因
- 如果首轮结果出现 `permission denied`、`Operation not permitted`、`access denied`、`docker.sock` 权限错误，不要把它误判成“服务不存在”；先判断是否需要带 `sudo` 的只读探测，并走审批

常见收敛结果：

- 入口层故障
- 业务进程 hang / 假活
- 容器异常 / OOM / CrashLoop
- 依赖异常
- 资源耗尽
- 配置/发布回归
- 网络 / 权限 / 证书问题
- 目标项目或目标服务锁错了

切专项技能的建议：

- 有 Docker / containerd 迹象 → `docker-container`
- 有 Pod / Deployment / Ingress / Service 迹象 → `kubernetes`
- 明显是 DNS / 端口 / 路由 / 证书 → `network` / `ssl-certificate`
- 明显是数据库或缓存 → `database`
- 明显是 Java 进程 → `jvm`

## 第五步：修复与验证

在故障域未收敛前，不要直接建议：

- `restart`
- `redeploy`
- `rollback`
- `kill`
- 修改配置

只有满足下面三个条件，才进入修复：

1. 已经锁定目标
2. 已经有证据链
3. 已经知道修复后怎么验证

修复后必须回到原始症状验证，例如：

- 用户说接口超时 → 修复后再次 curl / 探活
- 用户说页面报错 → 修复后验证页面请求恢复
- 用户说任务堆积 → 修复后验证任务恢复消费

## 输出要求

在思考时始终显式说明：

- 当前症状归类
- 当前候选目标
- 当前观测面
- 当前故障域是否已收敛
- 下一步为何选择该探针
