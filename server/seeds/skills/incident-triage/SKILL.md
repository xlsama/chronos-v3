## 第一步：症状归类与假设建立

从事件描述提炼"用户看到什么"，归入症状桶：

- 可用性：无响应、超时、5xx、健康检查失败
- 性能：慢、延迟高、CPU/内存高
- 数据正确性：查不到、重复、错乱
- 依赖异常：数据库、缓存、MQ、第三方不可用
- 变更异常：发布、配置、镜像变更后异常
- 安全/网络：DNS、证书、防火墙、鉴权失败
- 资源异常：磁盘满、FD 耗尽、OOM

基于症状提出 2-3 个假设（H1/H2/H3），按可能性排序，附空假设（可能是误报/瞬时问题/计划维护）。

调用 `update_plan` 写入初始调查计划，所有假设标记 `[待验证]`。

## 第二步：目标锁定

1. 优先从知识库上下文、AGENTS.md、事件描述中提取候选目标
2. 不足时调用 `list_servers()` + `list_services()` 补全
3. 整理出：候选项目、候选服务、候选服务器、关键依赖
4. 多候选时用区分线索收敛：域名、端口、容器名、IP、数据库名

## 第三步：逐层验证（循环）

取优先级最高的 `[待验证]` 假设，选择对应观测面：

| 症状指向 | 首选观测面 | 典型探针 |
|---------|-----------|---------|
| 无响应/超时/5xx | 外部探测 | curl、端口探活 |
| 进程挂/假活/资源耗尽 | 运行面 | docker ps、进程状态、top/free/df |
| 数据库/缓存/MQ 异常 | 依赖面 | service_exec 验活 |
| 需要错误栈/OOM/崩溃 | 日志面 | docker logs、应用日志 |
| 与发布/配置时间相关 | 变更面 | 最近发布记录、配置变更 |

每轮：

1. 执行探针，记录证据（正向/反向）
2. 更新假设状态：`[待验证]` → `[排查中]` → `[已确认]` 或 `[已排除]`
3. 调用 `update_plan` 同步假设状态和证据

循环退出：某个假设 `[已确认]` → 进入第四步；全部 `[已排除]` → 回到第一步重建假设。

## 第四步：修复与收尾

前置条件：目标已锁定 + 证据链已形成 + 知道如何验证。

修复后必须回到原始症状验证（接口超时 → curl 确认恢复；任务堆积 → 确认消费恢复）。

如故障域明确属于某专项，切换专项技能深入：

| 迹象 | 切换技能 |
|------|---------|
| Docker/容器 | docker-container |
| K8s/Pod/Deployment | kubernetes |
| DNS/端口/证书 | network / ssl-certificate |
| 数据库/缓存 | database |
| Java 进程 | jvm |
